Вывод
В ходе эксперимента с использованием Scapy был перехвачен HTTP POST-запрос, содержащий XSS-payload. В теле запроса присутствовал параметр snippet с внедрённым JavaScript-кодом, что подтверждает передачу вредоносного скрипта по сети в открытом виде. В ответ на запрос сервер вернул сообщение HTTP/1.0 501 Unsupported method ('POST'), что указывает на отсутствие поддержки метода POST, однако сам факт обработки и ответа сервера подтверждает успешную передачу запроса.

Во время XSS-атаки в сетевом трафике произошли следующие изменения: до атаки наблюдались только HTTP GET-запросы для загрузки веб-страницы, в момент атаки появился HTTP POST-запрос с пользовательскими данными и XSS-payload, после чего сервер отправил HTTP-ответ с кодом состояния 501. Таким образом, XSS-атака привела к появлению POST-запросов с вредоносным содержимым в трафике, что подтверждает возможность перехвата и анализа таких атак при использовании незащищённого HTTP-соединения.

Этап 5
В ходе работы были обнаружены уязвимости, связанные с XSS-атаками, обработкой пользовательского ввода и передачей данных по сети. Для их устранения рекомендуется внедрить следующие меры защиты.

1. Необходимо выполнять строгую фильтрацию и экранирование пользовательского ввода. Все данные, вводимые пользователем (формы New Snippet, Profile, Upload и другие), должны проходить серверную валидацию и HTML-экранирование, чтобы предотвратить выполнение внедрённого JavaScript-кода.
2. Следует использовать Content Security Policy (CSP). Настройка CSP позволяет ограничить выполнение встроенных скриптов и загрузку ресурсов только из доверенных источников, что значительно снижает риск эксплуатации XSS-уязвимостей.
3. Необходимо ограничить функционал загрузки файлов. Рекомендуется проверять MIME-тип и расширение загружаемых файлов, запрещать загрузку HTML и JavaScript файлов, а также хранить загруженные файлы вне web-директории, чтобы исключить их прямое выполнение браузером.
4. Следует использовать HTTPS для защиты сетевого трафика. Шифрование соединения предотвращает перехват и анализ HTTP-запросов и ответов, включая XSS-payload и cookies.
5. Рекомендуется использовать безопасные cookie (HttpOnly, Secure, SameSite), чтобы предотвратить их кражу через XSS-атаки.
6. Необходимо внедрить серверную проверку входных данных, ограничить длину вводимых параметров, использовать библиотеки для безопасной обработки HTML и регулярно проводить аудит безопасности веб-приложения.

Реализация данных мер значительно снизит риск эксплуатации XSS-уязвимостей и повысит общий уровень безопасности веб-приложения.
